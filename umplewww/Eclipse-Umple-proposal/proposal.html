The Umple Model-Oriented Programming project is a proposed open source project under the Modeling Container Project.

This proposal is in the Project Proposal Phase (as defined in the Eclipse Development Process) and is written to declare its intent and scope. We solicit additional participation and input from the Eclipse community. Please send all feedback to the Eclipse Proposals Forum.

Background

Scope

The Umple project encompasses in short, a model compiler and related techology: Specifically: The textual programming-language-like syntax called Umple, covering UML constructs, including associations, attributes, state machines, components, consraints, concurrency, etc. Real-time profiles such as Marte and AutoSar Patterns Separation of concerns in the context of Umple including mixins, aspect-oriented code injections, and variability modeling constructs A tracing sublanguage (MOTL) allowing injection of tracepoints into modeling constructs Technology for embedding Umple in programming languages such as Java, PhP, Ruby and C++ Technology for embedding code in such programming languages, plus Alf, into Umple A grammar DSL optimized for processing Umple and the embedding of languages, but also capable of being used in other contexts A subsystem for analysing the syntactic and semantic validity of models in Umple Code generation capabilities related to Umple A generic code-generation architecture designed to work with Umple, but also capable of working with other modeling front-ends Specific code generators for Java, PhP, Ruby, C++, covering all the syntactic elements described above Generators for creating UML diagrams from Umple Generators for exporting other UML representations (XMI, Yuml, etc.), SDL and SQL Generators for metrics and other data about Umple systems Generators for formal languages An Eclipse plugin supporting all of the above An XText-based plugin to give Umple improved syntax-directed editing, and a tool to convert the Umple grammar to XText A website (UmpleOnline) for editing Umple and UML diagrams interchangeably and generating code and other artifacts Metadata for the Umple user manual, and a tool to generate the user manual from this metadata Tools to generate self-documenting elements of the Umple compiler A User-Interface generation tool (UIGU) to generate usable default websites from any given Umple model for rapid prototyping A tool to facilitate transformation of code to Umple models+code (umplification) An extensive suite of testcases to test the syntax, Umple metamodel, and generated code Example systems written in Umple to serve as additional tests Plugins to various external editors to provide those editors with an Umple profile

Description

Why Eclipse?

Umple has been developed as a stand-alone project over the last five years, with the sponsorship of IBM and contributions to support the development of various components by other companies. Although rigorous quality has been built into the compiler using test-driven development and by writing the compiler in itself, we need the visibility and IP policies that an organization like Eclipse can provide, in order to help foster industrial uptake. Since Umple has been developed in Eclipse, and has an Eclipse plugin as one of its main artifacts, the Eclipse foundation seems the right home. Also according to XXXX Eclipse is one of the most open of open-source initiatives. Umple has always prided itself in being extremely open, and we want this to continue.

Initial Contribution

Umple has been an open source project under Google Code since 2011. It is licensed under the MIT license. Copyright is jointly held by the contributors, who, by virtue of agreeing to release their contributions under the MIT license have allowed their work to be freely used.

Legal Issues

There should be no important legal issues that we are aware of. The project is under the MIT license. We have no objection to converting to the Eclipse Public License going forward. The project does have dependencies on some other Eclipse projects (e.g. Jet). The website also uses CodeMirror and JQuery which are licensed under the MIT license. There is the issue of the generated code. When code is emitted it no license is injected, Our concept is that the original model can have whatever license users want, and the generated code would have the same license.
